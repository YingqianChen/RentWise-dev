## 2. 技术栈详解

### 2.1 Frontend 技术栈

| 技术 | 版本 | 作用 | 选择原因 |
|------|------|------|----------|
| **Next.js** | 14.2+ | React 框架 | 支持 SSR（服务端渲染）、文件路由、API 路由 |
| **React** | 18.2+ | UI 组件库 | 组件化开发、虚拟 DOM、生态成熟 |
| **TypeScript** | 5.0+ | 类型系统 | 静态类型检查、更好的 IDE 支持、减少运行时错误 |
| **TailwindCSS** | 3.4+ | CSS 框架 | 原子化 CSS、快速开发、一致性设计 |
| **SWR** | 2.2.5 | 数据获取 | 缓存、自动重新验证、错误重试 |

#### Next.js 的优势

```
传统 React SPA                    Next.js App Router
─────────────────                 ─────────────────────
客户端渲染                        服务端渲染 (SSR) 支持
需要额外配置路由                   文件系统路由（自动）
SEO 困难                          SEO 友好
首屏加载慢                        优化首屏性能
```

### 2.2 Backend 技术栈

| 技术 | 版本 | 作用 | 选择原因 |
|------|------|------|----------|
| **FastAPI** | 0.109+ | Web 框架 | 异步支持、自动 API 文档、类型提示 |
| **Python** | 3.10+ | 编程语言 | AI/ML 生态丰富、开发效率高 |
| **SQLAlchemy** | 2.0+ | ORM | 异步支持、成熟稳定、灵活查询 |
| **Alembic** | 1.13+ | 数据库迁移 | 版本控制数据库 schema |
| **Pydantic** | 2.7+ | 数据验证 | 与 FastAPI 完美集成、类型安全 |
| **asyncpg** | - | PostgreSQL 驱动 | 异步数据库操作 |

#### FastAPI 的优势

```
传统 Flask/Django                FastAPI
─────────────────                ─────────────────────
同步请求                         原生异步支持
手动 API 文档                    自动生成 Swagger 文档
无类型提示                       强类型系统
性能一般                         高性能（接近 Node.js）
```

### 2.3 AI/ML 技术栈

| 技术 | 作用 | 选择原因 |
|------|------|----------|
| **LangChain** | LLM 应用框架 | 模块化组件、工具集成 |
| **LangGraph** | Agent 工作流 | 状态管理、复杂流程编排 |
| **PaddleOCR** | OCR 文字识别 | 中文支持好、开源免费 |
| **Groq API** | LLM 云服务 | 快速响应、Llama 模型 |
| **Ollama** | 本地 LLM | 隐私保护、无需 API Key |

### 2.4 Database 技术栈

| 技术 | 作用 | 选择原因 |
|------|------|----------|
| **PostgreSQL** | 关系型数据库 | ACID 事务、复杂查询、成熟稳定 |

#### 为什么选择 PostgreSQL 而非 SQLite？

| 特性 | PostgreSQL | SQLite |
|------|-----------|--------|
| 并发写入 | ✅ 支持 | ❌ 锁定整个数据库 |
| 数据类型 | 丰富（JSONB、数组等） | 基础 |
| 生产部署 | ✅ 适合 | 仅适合开发/测试 |
| 扩展性 | 高 | 低 |

## 4. 关键技术决策

### 4.1 为什么选择 LangGraph？

**问题**：简单的 LLM API 调用不足以处理复杂的决策流程。

**LangGraph 的优势**：

```
简单 API 调用                      LangGraph Agent
─────────────────                  ─────────────────────
单次请求-响应                       多步骤状态机
无状态管理                         内置状态管理
难以处理复杂逻辑                    可视化流程图
难以调试                           支持断点、重试
无法中途干预                        支持人工干预
```

**RentWise 中的 LangGraph 应用**：
- Investigation Graph：调查问题生成流程
- Compare Agent：对比分析流程
- 状态追踪和错误恢复

### 4.2 为什么使用后台任务处理？

**问题**：OCR 和 LLM 分析耗时较长，同步处理会导致请求超时。

**解决方案**：FastAPI BackgroundTasks

```
同步处理                          异步后台处理
─────────────────                  ─────────────────────
用户等待 30-60 秒                   用户立即得到响应
请求可能超时                        页面显示"处理中"
用户体验差                          用户可以浏览其他内容
服务器压力大                        服务器负载分散
```

**处理状态机**：
```
queued → running_ocr → extracting → completed
                            │
                            ▼
                          failed (记录错误原因)
```

### 4.3 为什么选择 JWT 认证？

**JWT (JSON Web Token) 的优势**：

```
Session-based 认证                JWT 认证
─────────────────                  ─────────────────────
服务器存储 Session                  无状态，服务器不存储
需要 Session 存储                   无需额外存储
跨服务困难                          易于微服务扩展
CSRF 风险                          无 CSRF 风险
```

**RentWise 的 JWT 实现**：
- 7 天有效期
- 存储在浏览器 localStorage
- 每次请求通过 `Authorization: Bearer <token>` 发送

## 6. 环境配置

### 6.1 Backend 环境变量

```bash
# 必需配置
SECRET_KEY=your-secret-key-at-least-16-characters
DATABASE_URL=postgresql+asyncpg://user:password@host:5432/rentwise
LLM_PROVIDER=groq  # 或 ollama

# LLM 配置 (选择其一)
# Groq (云服务)
GROQ_API_KEY=your-groq-api-key
GROQ_MODEL=llama-3.3-70b-versatile

# Ollama (本地)
OLLAMA_HOST=http://localhost:11434
OLLAMA_MODEL=llama3.3

# OCR 配置
OCR_PROVIDER=paddleocr
PADDLEOCR_LANG=ch

# 文件存储
FILE_STORAGE_PROVIDER=local
LOCAL_UPLOAD_ROOT=./storage
```

NEXT_PUBLIC_API_URL=http://localhost:8000
