## 2. 各部分要点

### 2.1 开场：问题与解决方案 (1-2 分钟)

**开场白建议**：
> "在香港租房是一件充满挑战的事情——信息过载、隐性成本、条款复杂。我们团队开发了 RentWise，一个 AI 驱动的租房决策助手，帮助用户系统化地分析房源、识别风险、做出明智决策。"

**核心问题**：
- 信息过载：大量房源，难以比较
- 隐性成本：管理费、差饷、押金等
- 决策困难：不知道下一步该做什么

**解决方案**：
- 自动信息提取和结构化
- AI 风险评估
- 智能对比和行动建议

---

### 2.2 技术架构 (3-4 分钟)

**架构图展示要点**：

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  Frontend   │ ──▶ │   Backend   │ ──▶ │  Database   │
│  (Next.js)  │     │  (FastAPI)  │     │ (PostgreSQL)│
└─────────────┘     └─────────────┘     └─────────────┘
                           │
                           ▼
                    ┌─────────────┐
                    │ AI Services │
                    │ OCR + LLM   │
                    └─────────────┘
```

**技术选型理由**：

| 技术 | 选择原因 |
|------|----------|
| **Next.js** | 现代 Web 框架，支持 SSR，SEO 友好 |
| **FastAPI** | Python 异步框架，AI/ML 生态丰富 |
| **PostgreSQL** | 成熟的关系型数据库，支持复杂查询 |
| **LangGraph** | Agent 工作流编排，状态管理 |

**重点强调**：
- 前后端分离架构
- 异步后台处理（不阻塞用户）
- 模块化服务设计

### 2.3 AI 功能实现 ⭐ 核心亮点 (4-5 分钟)

**这是汇报的重点，建议详细展开**

#### 2.3.1 多模态输入处理

**要点**：
- 用户可以输入文字 + 上传图片
- 图片通过 OCR 自动识别文字
- 所有文本合并后送入 LLM 分析

**技术亮点**：
- PaddleOCR 中文支持优秀
- 异步 OCR 处理不阻塞用户

#### 2.3.2 LLM 信息提取

**要点**：
- 从非结构化文本提取结构化数据
- 提取 15+ 个关键字段（租金、面积、位置等）
- 处理"未知"情况，避免猜测

**展示 Prompt 设计**：
```
Extract the key rental fields from the Hong Kong listing text.
If a field is missing, return "unknown".
```

#### 2.3.3 三层评估体系 ⭐

**这是最重要的技术亮点**

```
┌─────────────────────────────────────────────────────────┐
│                    三层评估体系                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   1️⃣ Cost Assessment (成本评估)                         │
│      • 计算月总成本                                      │
│      • 识别隐性费用风险                                  │
│      • 输出: 成本置信度 + 风险标记                       │
│                                                         │
│   2️⃣ Clause Assessment (条款评估)                       │
│      • 分析维修责任                                      │
│      • 评估租期稳定性                                    │
│      • 输出: 条款风险标记                                │
│                                                         │
│   3️⃣ Candidate Assessment (整体评估)                    │
│      • 综合所有信息                                      │
│      • 生成下一步行动建议                                │
│      • 输出: 潜在价值 + 决策风险 + 行动建议              │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**关键概念解释**：
- `potential_value_level`: 这个房源值得花时间吗？
- `completeness_level`: 信息够不够做决策？
- `next_best_action`: 用户下一步该做什么？

#### 2.3.4 LangGraph Agent 工作流 ⭐ 创新点

**强调这是创新点**：

> "传统的 LLM API 调用无法处理复杂的多步骤流程。我们使用了 LangGraph，一个基于状态机的 Agent 框架，来管理复杂的决策流程。"

**LangGraph 优势**：
- 状态管理：自动追踪中间状态
- 流程可视化：清晰的处理节点
- 可扩展：易于添加新节点

**对比说明**：

| 传统方法 | LangGraph Agent |
|----------|-----------------|
| 单次请求-响应 | 多步骤状态机 |
| 无状态管理 | 内置状态管理 |
| 难以调试 | 可检查每步状态 |

#### 2.3.5 对比决策系统

**要点**：
- 不是简单的排序，而是"决策分组"
- 四个分组：最佳选择、备选、信息不足、建议放弃
- 每个分组有解释和行动建议

## 3. 可能的提问与回答

### Q1: 为什么选择 PostgreSQL 而不是 MongoDB？

**回答要点**：
- 数据关系复杂（用户→项目→候选→评估）
- 需要 ACID 事务保证数据一致性
- PostgreSQL 的 JSONB 类型可以处理半结构化数据

### Q2: LangGraph 和直接调用 LLM API 有什么区别？

**回答要点**：
- LLM API 是单次请求-响应
- LangGraph 是状态机，可以管理多步骤流程
- 支持状态追踪、错误恢复、流程可视化

### Q3: 如何处理 OCR 识别错误？

**回答要点**：
- OCR 结果保留为证据，用户可以查看
- 后续 LLM 提取会综合判断
- 用户可以在候选详情页编辑修正

### Q4: 评估算法的准确性如何保证？

**回答要点**：
- 使用置信度标记（high/medium/low）
- 关键决策需要用户核实
- 不隐藏不确定性，明确告知用户

### Q5: 这个系统如何部署？

**回答要点**：
- Frontend 可部署到 Vercel/Netlify
- Backend 可部署到云服务器（AWS/GCP）
- Database 可使用托管服务（Neon/Supabase）

### Q6: 如果 LLM 服务不可用怎么办？

**回答要点**：
- 支持本地 Ollama 作为备用
- 关键功能有规则基础的 fallback
- 用户仍可手动录入和编辑信息

## 4. 项目局限性讨论

**诚实面对局限性**：

| 局限 | 原因 | 改进方向 |
|------|------|----------|
| **市场数据有限** | SDU 基准数据范围有限 | 扩展数据来源 |
| **不支持通勤计算** | 需要位置模型和地图 API | Phase 3 计划 |
| **对比结果不持久化** | 实时生成，历史丢失 | 添加保存功能 |
| **单语言界面** | 目前仅英文界面 | 国际化支持 |
